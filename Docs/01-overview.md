# 01 — 全景总览：什么是 Agentic 系统

> 学习笔记 · 基于 nanobot 项目

---

## 一句话定义

**Agentic 系统 = LLM + 工具调用 + 记忆 + 自主决策循环**

普通聊天机器人只能"一问一答"（text-in-text-out），而 Agentic 系统能**自主决定下一步做什么**：要不要查文件？要不要执行命令？需不需要上网搜索？什么时候该停下来回复用户？

这就像一个实习生 vs 一个老员工的区别——实习生只会回答问题，老员工会主动拆解任务、查资料、找人协作、最后给你结果。

---

## nanobot 的整体架构

nanobot 的架构可以用一张图说清楚：

```
┌──────────────────────────────────────────────────┐
│              用户层（Channel）                      │
│     Telegram / Slack / Discord / CLI / ...        │
└────────────────────┬─────────────────────────────┘
                     │ ① 用户发消息
                     ▼
┌──────────────────────────────────────────────────┐
│              消息总线（MessageBus）                  │
│          入站队列 ←──────→ 出站队列                  │
└────────┬───────────────────────────┬─────────────┘
         │ ② Agent 取消息            │ ⑥ 回复推给渠道
         ▼                          ▲
┌──────────────────────────────────────────────────┐
│              Agent 核心引擎                        │
│                                                  │
│   ┌──────────────────────────────────────────┐   │
│   │  AgentLoop — ReAct 循环（核心中的核心）       │   │
│   │  while 没完成:                             │   │
│   │      ③ 问 LLM："下一步做什么？"              │   │
│   │      ④ LLM 说"调用 xx 工具" → 执行工具       │   │
│   │      ⑤ 把工具结果告诉 LLM → 继续循环         │   │
│   │  LLM 说"我想好了" → 输出最终回答              │   │
│   └──────────────────────────────────────────┘   │
│                                                  │
│   配套模块：                                      │
│   · ContextBuilder — 组装 system prompt           │
│   · ToolRegistry   — 管理所有工具                  │
│   · MemorySystem   — 长短期记忆                    │
│   · SubagentManager — 子 Agent 协作               │
│   · SessionManager  — 会话持久化                   │
└──────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────┐
│              LLM 抽象层（Provider）                 │
│     OpenAI / Anthropic / DeepSeek / 本地模型 ...   │
└──────────────────────────────────────────────────┘
```

## 五大核心模块

| 模块 | 一句话说明 | 对应源码 |
|------|-----------|---------|
| **消息总线** | 渠道和 Agent 之间的"传话筒"，双向异步队列 | `bus/queue.py`、`bus/events.py` |
| **Agent 循环** | ReAct 循环：思考 → 调工具 → 看结果 → 再思考 | `agent/loop.py` |
| **工具系统** | Agent 能用的"技能"：读文件、跑命令、搜网页等 | `agent/tools/` 目录 |
| **记忆系统** | 短期记忆（会话历史）+ 长期记忆（MEMORY.md） | `agent/memory.py`、`session/manager.py` |
| **多 Agent 协作** | 主 Agent 派活给子 Agent，后台并行干活 | `agent/subagent.py` |

## 一条消息的完整生命周期

用一个具体例子来说：用户在 Telegram 里说"帮我看看 main.py 有没有 bug"

```
1. Telegram 渠道收到消息
   → 封装成 InboundMessage（谁发的、哪个群、说了什么）
   → 丢进 MessageBus 的入站队列

2. AgentLoop 从入站队列取到这条消息
   → 加载这个用户的会话历史（Session）
   → 用 ContextBuilder 拼接 system prompt + 历史 + 当前消息
   → 组装成完整的 messages 数组

3. 进入 ReAct 循环
   → 第1轮：LLM 说"我要调用 read_file 看 main.py"
            执行 read_file 工具，拿到文件内容
   → 第2轮：LLM 看了文件内容，说"第42行有个空指针风险"
            不再调用工具 → 循环结束

4. 把最终回答封装成 OutboundMessage
   → 丢进 MessageBus 的出站队列

5. ChannelManager 从出站队列取到回复
   → 路由给 Telegram 渠道
   → Telegram 发给用户
```

## 为什么要分这么多层？

| 设计决策 | 好处 | 如果不这样做 |
|---------|------|------------|
| 消息总线解耦渠道和 Agent | 加一个新渠道（如钉钉）不用改 Agent 代码 | 每加一个渠道就要改 Agent，耦合爆炸 |
| 工具用 Registry 统一管理 | 加一个新工具只需继承基类 + 注册 | 每加一个工具要改 AgentLoop 代码 |
| LLM 抽象成 Provider 接口 | 换模型只需改配置，一行代码不用动 | 换模型要大改调用逻辑 |
| 会话持久化到文件 | 重启不丢对话，每个用户的聊天独立 | 一重启啥都忘了 |

## 和市面上其他框架的对比

| 特性 | nanobot | LangChain | AutoGPT |
|------|---------|-----------|---------|
| 核心模式 | ReAct 循环 | Chain/Agent 可选 | 任务规划 + 执行 |
| 工具扩展 | 简单（继承 Tool 基类） | 丰富但复杂 | 插件系统 |
| 记忆 | 双层（Session + MEMORY.md） | 多种可选 | 向量数据库 |
| 多渠道 | 内置 10+ 渠道 | 需自己接 | 无 |
| 代码复杂度 | ~5000行核心代码 | 数万行 | 数万行 |
| 适合学习 | ✅ 非常适合 | ❌ 太复杂 | ❌ 太复杂 |

> **nanobot 的最大价值是"麻雀虽小五脏俱全"**——用最少的代码实现了一个完整的工业级 Agentic 系统，非常适合学习。

## 面试话术

> "Agentic 系统和普通 Chatbot 的本质区别在于**自主决策能力**。普通 Chatbot 是 text-in-text-out，而 Agent 能自主决定是调用工具还是直接回答。核心架构通常由 5 部分组成：消息接入层、ReAct 循环引擎、工具系统、记忆系统和 LLM 抽象层。各层之间通过消息总线和注册表模式解耦，实现了高度可扩展性。"
